## 🏗️ АРХИТЕКТУРА — Превосходная

### ✅ Что сделано отлично

**Clean Architecture с чётким разделением слоёв:**
```
data/hal/          → Аппаратная абстракция (BLE/USB/Mock)
domain/math/       → Чистая математика (фильтры, LTTB)
domain/entities/   → Бизнес-объекты (без зависимостей)
presentation/      → UI + Riverpod-провайдеры
```

**Паттерн Composite в SensorHub** (837 строк) — управляет N USB-устройствами одновременно. Горячее подключение/отключение, auto-detect через реестр Windows, экспоненциальный backoff при сбоях переподключения (3с → 60с потолок) — всё реализовано на уровне промышленного ПО.

**HAL-интерфейс** — единый `Stream<SensorPacket>` для BLE, USB, Mock. Можно менять транспорт без изменения бизнес-логики.

**Обработка сигналов** — мирового уровня:
- **1€ Filter** (стандарт Apple/Google/Microsoft) для дальномера
- **Фильтр Калмана** с попараметральной настройкой под каждый датчик
- **SpikeGuard** — отсекает выбросы
- **LTTB** — 100K точек → 1K без потери формы графика

## 🔴 КРИТИЧЕСКИЕ ПРОБЛЕМЫ (Серьёзность: КРИТИЧЕСКАЯ)

### C1. Прошивка ESP32: все драйверы датчиков — заглушки

**Файл**: `firmware/src/main.cpp`, строки 350-430

Все 6 драйверов датчиков (INA226, BME280, LSM6DS3, MAX31855, VL53L1X) помечены `// TODO: Read real sensor` и генерируют случайные данные через `sin()` + `random()`:

```cpp
// ❌ Текущее состояние (для ВСЕХ датчиков):
packet.voltage_v = 3.3 + 0.5 * sin(millis() / 1000.0);  // TODO: Read INA226
packet.temperature_c = 25.0 + 5.0 * sin(millis() / 5000.0);  // TODO: Read BME280
```

**Влияние**: Мультидатчик **не работает с реальным оборудованием** — только симуляция. Это блокер для любого физического использования.

**Рекомендация**: Приоритет №1. Реализовать I2C-драйверы для каждого чипа. Библиотеки ESP-IDF уже доступны. Ориентировочный объём работы — 200-300 строк на датчик.

### C2. Отсутствие валидации CRC на стороне прошивки для входящих команд

**Файл**: `firmware/src/main.cpp`, строки 245-290

Прошивка формирует BLE-пакет из 84 байт (MAGIC + данные + CRC8), и CRC проверяется на стороне приложения при приёме (`ble_hal.dart`). Однако **командный канал** (запись из приложения → ESP32) **вообще не защищён CRC**. Повреждённая команда START/STOP/CALIBRATE будет выполнена без проверки.

**Влияние**: При BLE-помехах возможна ложная остановка или запуск эксперимента.

**Рекомендация**: Добавить CRC-валидацию входящих команд в `onWrite` callback прошивки.

## 🟠 ВЫСОКИЙ ПРИОРИТЕТ (Серьёзность: ВЫСОКАЯ)

### H1. Утечка ресурсов: Timer горячего подключения в SensorHub не останавливается

**Файл**: `lib/data/hal/sensor_hub.dart`, строки 185-220

```dart
void _startHotPlugMonitor() {
  _hotPlugTimer = Timer.periodic(Duration(seconds: 3), (_) async {
    await _checkForNewDevices();
  });
}
```

`_hotPlugTimer` запускается при `connect()`, но **не отменяется** в `disconnect()` при определённых условиях (если `_isDisconnecting` уже `true` и срабатывает ранний `return`). Таймер продолжает сканировать реестр Windows каждые 3 секунды после отключения.

**Влияние**: Бесполезная нагрузка на слабых школьных ПК (Celeron N4000). При многократном переподключении — множественные параллельные таймеры.

**Рекомендация**: Гарантировать `_hotPlugTimer?.cancel()` в **начале** `disconnect()`, до любых проверок `_isDisconnecting`.

---

### H2. Гонка состояний (Race Condition) при быстром переподключении USB

**Файл**: `lib/data/hal/usb_hal_windows.dart`, строки 290-350

```dart
Future<void> connect(String portName) async {
  if (_isConnecting) return;  // Защита от двойного вызова
  _isConnecting = true;
  // ... (длительная операция: probe + open + настройка)
  _isConnecting = false;
}
```

Флаг `_isConnecting` — не атомарный. Если два вызова `connect()` происходят одновременно из разных провайдеров (SensorHub + ручное подключение), оба могут пройти проверку `if (_isConnecting)` до установки `true`.

**Влияние**: Два процесса пытаются открыть один COM-порт → `errno 5 (Access Denied)` → бесконечные retry. На практике вероятность низкая, но при горячем подключении устройства возможна.

**Рекомендация**: Использовать `Completer` или `Lock` из пакета `synchronized` для гарантии эксклюзивного доступа.

---

### H3. Буфер эксперимента — обрезка 10% при переполнении теряет данные без предупреждения

**Файл**: `lib/presentation/blocs/experiment/experiment_provider.dart`, строки 580-600

```dart
if (_data.length > 500000) {
  _data = _data.sublist(_data.length ~/ 10);  // Обрезаем 10%
}
```

При превышении 500K точек обрезаются **первые 10% данных** (50K точек). Пользователь **не уведомляется**, и обрезанные данные **невосстановимы** — не сохраняются в SQLite.

**Влияние**: На длительных экспериментах (>14 часов при 10 Гц) учитель теряет начало эксперимента без предупреждения. На слабых ПК с большими частотами — быстрее.

**Рекомендация**:
1. Показать пользователю предупреждение при достижении 80% буфера
2. Автоматически сохранять обрезаемые данные в SQLite (Drift)
3. Использовать кольцевой буфер вместо `List` + `sublist` (O(n) копирование)

---

### H4. BLE-переподключение: жёсткий лимит 5 попыток без восстановления

**Файл**: `lib/data/hal/ble_hal.dart`, строки 200-250

```dart
if (_reconnectAttempts >= _maxReconnectAttempts) {  // _max = 5
  _updateState(HalConnectionState.error);
  return;  // Сдаёмся навсегда
}
```

После 5 неудачных попыток переподключения BLE устройство переходит в состояние `error` **навсегда**. Единственный способ восстановить связь — перезапустить приложение.

**Влияние**: В школе, где BLE-помехи от 30 телефонов учеников — частая ситуация. Учитель не поймёт, почему приложение перестало видеть датчик.

**Рекомендация**: 
1. Добавить кнопку «Переподключить» в UI при состоянии `error`
2. Сбрасывать счётчик `_reconnectAttempts` при ручном вызове `connect()`
3. Увеличить лимит до 10 с нарастающим backoff (уже частично реализован)

---

### H5. Дублирование логики сканирования портов

**Файлы**: `lib/data/hal/port_scanner.dart` (строки 50-150) и `lib/data/hal/port_connection_manager.dart` (строки 80-170)

Оба класса содержат **практически идентичную** логику сканирования COM-портов через `reg query HKLM\HARDWARE\DEVICEMAP\SERIALCOMM`. `PortScanner` — старый вариант с Isolate-сканированием, `PortConnectionManager` — новый вариант.

**Влияние**: Нарушение принципа DRY. Исправление бага сканирования в одном классе не распространяется на другой.

**Рекомендация**: Удалить `PortScanner` или извлечь общую логику сканирования реестра в отдельный утилитный класс `RegistryPortEnumerator`.

### M3. Отсутствие таймаута при открытии COM-порта через FFI

**Файл**: `lib/data/hal/port_connection_manager.dart`, строки 200-280

Метод `connect()` выполняет `Process.run("mode", [portName])` с таймаутом 3 секунды (✅), но последующее **открытие порта через FFI** (`SerialPort(portName)`) не имеет общего таймаута. Если `openReadWrite()` зависнет (редко, но возможно на повреждённом USB-драйвере), вызов заблокируется навсегда.

**Рекомендация**: Обернуть открытие порта в `Future.timeout(Duration(seconds: 5))`.

### M4. Отсутствие валидации timestampMs в ExperimentController

**Файл**: `lib/presentation/blocs/experiment/experiment_provider.dart`, строки 530-560

```dart
void _onPacket(SensorPacket packet) {
  if (!_isRunning) return;
  // Пакет с timestamp=0 добавится в _data без валидации
  _data.add(packet);
}
```

Пакеты с `timestampMs == 0` (битый пакет / потеря синхронизации) добавляются в буфер и отображаются на графике, создавая артефакты (скачок к началу оси X).

**Рекомендация**: Фильтровать пакеты с `timestampMs <= 0` или с `timestampMs < lastTimestamp` (нарушение монотонности).

### M5. Нет сохранения данных при аварийном завершении эксперимента

**Файл**: `lib/presentation/blocs/experiment/experiment_provider.dart`

При вызове `stopExperiment()` данные сохраняются. Но если приложение **крашится** или пользователь **закрывает окно** во время эксперимента, все данные из `_data` (List в памяти) **теряются безвозвратно**.

**Влияние**: Ученик провёл 40-минутную лабораторную → Windows обновился / зависло → все данные потеряны.

**Рекомендация**: Реализовать периодический автосейв в SQLite (каждые 30 секунд или каждые 1000 точек) + восстановление из автосейва при запуске.

## 📝 ЗАКЛЮЧЕНИЕ

Проект «ЛАБОСФЕРА» демонстрирует **высокий уровень инженерной культуры**. Архитектура, обработка сигналов и решение USB-проблем на Windows — на уровне старших инженеров ведущих компаний.

**Главный блокер**: прошивка ESP32 работает только в режиме симуляции — все драйверы датчиков являются заглушками. Без их реализации устройство не может использоваться с реальным оборудованием.

**Для школьных ПК**: приложение **готово к работе** на Celeron N4000 / 4 ГБ RAM. Все критические оптимизации производительности (LTTB, Isolate, адаптивный FPS, windowed rendering) уже реализованы.

**Рекомендуемый порядок действий**:
1. Реализовать I2C-драйверы датчиков в прошивке
2. Исправить утечку Timer и гонку состояний
3. Добавить автосейв эксперимента
4. Покрыть UI widget-тестами
5. Провести нагрузочное тестирование на целевом ПК

> **Вердикт**: При исправлении критических и высокоприоритетных проблем (Фазы 1-2, ~2-3 недели), проект будет готов к пилотному запуску в школах.
